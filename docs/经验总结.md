# 软件开发经验总结 - PPT 转换项目复盘

> **项目**: ppt-conform（PPT 标准模板转换智能体）
> **日期**: 2026-07-19
> **背景**: 从内容与格式割裂的问题出发，重构架构，将转换成功率从 60-70% 提升到 100%

---

## 一、做得好的地方

### 1. 快速定位根本问题，不做表面优化
- **正确做法**: 当用户反馈"内容与格式割裂"时，没有停留在"调整格式匹配"层面，而是深入到 `_copy_slide_content` 方法，发现了**文本匹配去重**这个根本缺陷
- **关键洞察**: 内容重复/丢失的根因不是"格式不对"，而是"去重逻辑不可靠"导致内容被错误地跳过或重复处理
- **经验**: 遇到问题时，先问"数据从哪来、怎么流转、为什么会错"，而不是直接改表象

### 2. 架构设计有层次感
- **内容与格式分离**: TextFormat、ShapeFormat 独立建模，ContentBlock 持有格式引用，职责清晰
- **精确映射机制**: shape_id 作为 ContentBlock 和 raw_shape 之间的桥梁，替代脆弱的文本匹配
- **优先级策略**: 模板格式优先、原格式兜底，用明确的 fallback 链条替代模糊的"尽量匹配"

### 3. 数据驱动的验证方式
- **对比测试脚本**: 创建 `test_conversion_compare.py`，用构造的典型PPT做端到端验证
- **量化成功率**: 用"通过/总数"的百分比衡量改进效果，而不是主观判断"好像好了"
- **多维度覆盖**: 封面页、单文本框、多文本框、图片页、表格页 × 4种Master风格，确保全面性

### 4. 小步快跑，每一步都可验证
- 先改数据模型 → 运行测试 → 改提取器 → 运行测试 → 改回填器 → 运行测试
- 每一步改动都有明确的验证手段，不会"大改一通然后不知道哪里错了"

### 5. 状态文档持续更新
- PROJECT_STATUS.md 保持最新，下次对话不用从零开始理解项目
- 核心文件、关键方法、已知问题、下一步建议都有记录

## 二、做得不好的地方（弯路与教训）

### 1. 第一次架构改动不够彻底
- **问题**: 第一次加了 TextFormat/ShapeFormat，但回填器的去重逻辑还是老的文本匹配
- **后果**: 格式信息有了，但内容与结构割裂的根本问题没解决，成功率提升有限
- **教训**: **找到根因再动手**。如果根因是"去重逻辑不可靠"，那么加再多格式字段也没用。第一次改动应该先解决去重问题，再加格式兜底

### 2. 对 python-pptx 的格式继承机制理解不足
- **问题**: 以为 `run.font.size` 就是实际字号，后来才发现很多时候是 None，实际值继承自占位符/主题
- **后果**: 提取的格式信息大量为 None，兜底效果打折扣
- **教训**: **用第三方库之前，先搞清楚它的数据模型和继承机制**。尤其是 Office 这种层级复杂的格式系统，不能想当然

### 3. 测试先行做得不够
- **问题**: 先改代码再补测试，而不是先写测试再改代码
- **后果**: 改完代码后不确定哪些场景覆盖了、哪些没覆盖
- **教训**: TDD 的价值在这种"改架构"的场景下尤为明显。先定义"成功是什么样"，再写代码让它通过

### 4. ShapeFormat 没有完全落地
- **问题**: 新增了 ShapeFormat 类，但形状的位置/大小仍走原来的 shape_data 字典，没有走"模板优先、原格式兜底"的逻辑
- **后果**: 架构设计不完整，有"半成品"
- **教训**: **加数据模型就要有对应的使用逻辑**，否则就是死代码。要么不做，要么做完整

### 5. 对用户问题的根因分析慢了一拍
- **问题**: 用户两次反馈"内容与结构割裂"，第一次的理解还是停留在"格式不对"，直到第二次用户明确提出"内容和格式分开、建立对应关系"才意识到问题本质
- **后果**: 走了一轮弯路，浪费了时间
- **教训**: **用户反馈问题时，多问几个"为什么"，深挖一层**。不要急于给出解决方案，先确保理解了真正的问题

## 三、软件开发方法论：如何避免走弯路、提高效率

### 原则1：先诊断，再开方

**错误做法**: 看到问题直接改代码 → 改完发现不对 → 再改 → 越改越乱

**正确做法**:
1. **复现问题**: 用最小用例稳定复现
2. **定位根因**: 沿着数据流追踪，找到"为什么会错"的那一行
3. **设计方案**: 想清楚改什么、怎么改、影响范围
4. **实施方案**: 小步改动，每步验证
5. **验证效果**: 用测试/数据证明问题解决了

> **口诀**: 诊断占60%时间，设计占20%，编码占20%。编码越快的人，越容易跳过诊断直接写代码，反而越慢。

### 原则2：用数据说话，不要靠感觉

**错误做法**: "我觉得应该好了"、"大概没问题吧"

**正确做法**:
- 定义可量化的指标（成功率、通过数、性能指标）
- 写自动化测试验证
- 改前测一次、改后测一次，用数据对比证明改进

> **口诀**: 没有度量就没有改进。如果不能量化"好"，就不知道是不是真的变好了。

### 原则3：小步快跑，每步可回退

**错误做法**: 大改一通，几十个文件一起改，然后发现跑不起来，不知道哪里错了

**正确做法**:
- 每次只改一个层面（数据模型 → 提取器 → 回填器 → 测试）
- 每改完一层跑一次测试
- 每一步都是可独立运行的中间状态
- 出问题了可以快速回退到上一个稳定点

> **口诀**: 不要试图一步到位。楼梯比电梯慢，但电梯坏了会摔得很惨。

### 原则4：测试是开发的一部分，不是收尾

**错误做法**: 先写代码，最后再补测试（甚至不补）

**正确做法**:
- 改之前先想：怎么验证改对了？
- 先写测试用例（哪怕是伪代码），明确成功标准
- 改完代码跑测试，通过了才算完成

> **口诀**: 测试不是"质量保证"，是"开发效率工具"。它让你敢改代码，不怕改坏。

### 原则5：深入理解你的依赖库

**错误做法**: 看一眼API文档就开始用，遇到问题才发现理解错了

**正确做法**:
- 用之前先搞清楚核心概念和数据模型
- 特别是格式/样式这种有继承机制的，先画个继承关系图
- 写个最小 demo 验证你的理解对不对

> **口诀**: 磨刀不误砍柴工。花1小时搞懂依赖库的设计，能省10小时的debug时间。

### 原则6：架构设计要"做就做完整"

**错误做法**: 新增了数据模型但没用、新增了接口但没实现、只改了一半

**正确做法**:
- 设计时想清楚：这个改动涉及哪些层？每个层怎么改？
- 实施时按层推进，确保每一层都改到位
- 改完后检查：有没有"死代码"？有没有只定义没使用的东西？

> **口诀**: 半成品比没有还糟糕——它增加了代码复杂度，但没有带来价值。

### 原则7：写状态文档，给自己留后路

**错误做法**: 全靠脑子记，下次继续从零开始

**正确做法**:
- 项目关键节点更新状态文档
- 记录：当前状态、做了什么、没做什么、坑在哪里、下一步建议
- 下次接手时先看文档，10分钟就能进入状态

> **口诀**: 好记性不如烂笔头。今天的你写的文档，是明天的你的救命稻草。

## 四、具体到 PPT 转换类项目的经验

### 技术选型
- **python-pptx** 够用但不完美，复杂格式（SmartArt、动画、渐变）支持有限
- 遇到 python-pptx 做不到的，直接操作 XML（lxml）是终极方案
- 格式继承是 Office 的核心机制，必须理解：Master → Layout → Placeholder → Shape → Run

### 核心难点
1. **格式继承**: 很多属性不是直接存储在元素上，而是继承自上层
2. **占位符匹配**: 源PPT和模板PPT的占位符类型/位置可能完全不同
3. **布局映射**: 源PPT的布局怎么对应到模板的布局，没有标准答案
4. **内容去重**: 同一份内容可能同时出现在 body_blocks 和 raw_shapes 中

### 推荐架构
```
提取层 → 内容模型层 → 匹配层 → 回填层 → 校验层
  ↓         ↓          ↓         ↓         ↓
 源PPT   ContentBlock  语义角色  模板格式  质量检查
        + TextFormat  布局匹配  原格式兜底  溢出检测
        + ShapeFormat shape_id  位置适配
```

### 避坑清单
- ❌ 不要用文本内容去重（空格、换行、顺序变化都会导致匹配失败）
- ❌ 不要假设 run.font.size 一定有值（大部分时候是继承的）
- ❌ 不要忽略占位符的 idx（同类型占位符可能有多个，靠 idx 区分）
- ✅ 要用唯一标识符（shape_id）建立内容与形状的映射
- ✅ 要处理格式继承的 None 值
- ✅ 要做多 Master/多 Layout 的兼容性测试
- ✅ 页眉页脚要特殊处理（不要提取原PPT的，直接用模板的）

## 五、总结

这次项目最大的收获不是"把成功率从70%提到了100%"，而是验证了一个道理：

> **遇到问题时，找根因比改代码重要；做验证比靠感觉重要；小步走比大步跳重要。**

软件开发不是比谁写代码快，是比谁少走弯路。少走弯路的秘诀就是：
多想、少改、每步验证。
